# Car Dealership
 Это информационная система для мультибрендового, частного, автосалона с широким спектром услуг. Проект охватывает полный цикл разработки современного backend-приложения: проектирование предметной области, реализацию бизнес-логики, REST API, работу с базой данных, безопасность, микросервисную архитектуру и межсервисное взаимодействие.


## Основной функционал

Система предоставляет возможности:

- просмотра и поиска автомобилей
- оформления заказа автомобиля из наличия
- конфигурирования автомобиля с индивидуальной комплектацией
- записи на тест-драйв
- управления каталогом автомобилей и запчастей
- сопровождения жизненного цикла заказов
- управления складскими процессами.

Поддерживаются роли пользователей:

- Клиент
- Менеджер автосалона
- Администратор склада
- Администратор системы.

# Этапы развития проекта

## Этап 1. Проектирование доменной модели

На первом этапе была реализована основная бизнес-логика системы:

- разработана доменная модель автосалона
- реализован конфигуратор автомобилей с проверкой совместимости комплектующих
- реализованы сценарии оформления заказов и тест-драйвов
- спроектирована гексагональная архитектура
- добавлены валидация данных и предметные исключения
- написаны unit-тесты.

---

## Этап 2. Построение backend-приложения

Проект был перенесен на стек Spring Boot.

Были реализованы:

- REST API
- PostgreSQL и Spring Data JPA
- миграции базы данных с Liquibase
- DTO и маппинг сущностей
- Swagger
- фильтрация данных через Spring Specifications
- интеграционные тесты с использованием Testcontainers.

---

## Этап 3. Реализация безопасности

Добавлены механизмы аутентификации и авторизации:

- Spring Security
- Keycloak
- ролевая модель доступа
- защита REST API
- проверка владельца заказа
- разграничение доступа пользователей согласно ролям.

---

## Этап 4. Переход на микросервисную архитектуру

Монолитное приложение было разделено на два независимых сервиса:

- **OrderService** — управление заказами и клиентскими сценариями
- **StorageService** — управление складом, автомобилями и комплектующими.

Также были реализованы:

- обмен сообщениями через Kafka
- событийная архитектура
- Outbox Pattern
- идемпотентная обработка сообщений
- отдельные базы данных и независимое развертывание сервисов.

---

## Этап 5. Межсервисное взаимодействие по gRPC

Для получения актуальной информации о наличии автомобилей реализовано синхронное взаимодействие между сервисами.

Добавлены:

- gRPC Server в StorageService
- gRPC Client в OrderService
- REST API для получения списка автомобилей
- обработка таймаутов и недоступности сервисов
- интеграционные тесты gRPC.
